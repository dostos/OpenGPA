# R13 Bevy code-only baseline — 2026-04-30

## Setup

5 Bevy scenarios mined in this round, all classed `framework-internal`,
SHA-pinned to the parent commit of the fix PR. Each scenario was patched
to include an `## Upstream Snapshot` section referencing
`https://github.com/bevyengine/bevy` at `fix_parent_sha`. Snapshots
pre-fetched into `/data3/opengpa-snapshots/`.

Each scenario was given to a fresh Claude Code Explore subagent with:

- the user-report sections only (`User Report`, `Expected Correct
  Output`, `Actual Broken Output`, `How OpenGPA Helps`) — Ground Truth,
  Fix, Bug Signature, and Upstream Snapshot blocks were stripped before
  prompting
- the snapshot path on disk
- read/list/grep access only — no run/build/network
- ~20-file investigation budget
- instruction to end with a fenced JSON block listing the bug-bearing
  files

Scoring: hit = the agent's `files[]` set intersects `expected_files`
from the scenario's `## Bug Signature` yaml.

## What this tells us

The Bevy scenarios are **too tractable** for code_only with a snapshot
in hand. Every bug had a clear keyword trail in the user report
(e.g. "depth prepass", "wireframe", "NoCpuCulling", "scrolling fog")
that mapped directly onto the file structure (`prepass/node.rs`,
`wireframe.rs`, `cluster/gpu.rs`, ...). With grep + read access to the
upstream tree, none of these required actual runtime evidence.

This means **these scenarios cannot demonstrate OpenGPA's marginal
value** — there is no code_only failure for OpenGPA to fix. They were
useful as smoke tests to verify that:

1. The Vulkan layer captures Bevy-style (wgpu-backed) Vulkan apps
   end-to-end — confirmed in `vulkan-shim-status.md`.
2. The `Upstream Snapshot` ingestion pipeline works for the
   `bevyengine/bevy` repo (snapshots fetched, files queryable).
3. Subagent-driven evals scale — 5 parallel agents finished in under
   90 seconds total.

## What would actually stress OpenGPA?

To produce a code_only failure mode that OpenGPA could rescue, future
scenarios should:

- **Hide the keyword trail.** Phrase the user report as observable
  visual symptoms only ("the red sphere is invisible when scene is
  cycled to all-transparent"), without naming the subsystem ("depth
  prepass"). The R10 / R11 three.js scenarios that beat code_only had
  this property — the report described shaders going dark, not the
  shader chunk by name.
- **Pick bugs whose location is not derivable from grep alone.** Bugs
  in cross-file ordering, missing render-pass dispatches, unintended
  pipeline state transitions — places where the *runtime trace* shows
  what *isn't* happening, while the static code looks fine.
- **Distribute symptoms over many candidate files.** If grepping
  "wireframe" returns 1 obvious file, the agent wins without OpenGPA.
  If it returns 30 files across crates, the agent needs frame-state
  evidence to choose.
