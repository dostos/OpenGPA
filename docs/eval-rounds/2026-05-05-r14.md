# Round R14 (2026-05-05)

Tested two prompt-level interventions on the same 14-scenario cohort:
P3-new (drop GPA tool block on browser-tier) and P2 (tool-call budget
hint). Mixed result: tier gate worked as designed; budget hint
over-throttled and cost solves.

## Ran

- Cohort: same 14 R12 scenarios (8 godot + 6 web-map)
- Modes: `with_bhdr`, `code_only`
- Model: `claude-opus-4-7[1m]` via claude-cli
- Output: `/data3/bhdr-eval-results/2026-05-05-r14/`
- Variables vs R13:
  - `tools["effective_mode"] = "code_only"` for browser-tier with_bhdr
    scenarios → no GPA tool block in the system prompt
  - "Stopping condition" paragraph added to all three bug-class
    prompts: "if you exceed ~20 tool calls without a clear
    hypothesis, emit best-effort low-confidence JSON and stop"

## Findings

- **Browser-tier gate works perfectly.** All 5 solvable web-map
  scenarios solved in BOTH modes (R13 had with_bhdr 5/6 vs code_only
  6/6 with the gap on translucency + edge_wall; R14 closed it). Only
  remaining web-map failure is `cesium_camera_jumps`, which is a
  scenario-quality issue (mining picked PR #12983; agent's correct
  diagnosis is for PR #13098 which addressed the same user complaint).
- **Budget hint over-throttled.** Multiple scenarios that previously
  solved at 15-20 tool calls now stop at 4-10 and emit low-confidence
  JSON that misses ground truth. Most striking:
  - `godot_glow_extremely_slow`: R13 wg-S/co-S → R14 wg-S/co-X. Code-only
    stopped at 8 tools, judge=partial.
  - `godot_web_images`: R13 wg-S/co-S → R14 wg-X/co-X. Both stopped at
    6-10 tools, both judge=partial.
  - `godot_weird_shadow` code_only: stopped at 4 tool calls (was
    33 in R12c).
- **Token spend dropped 13-15%** but at -2/-4 solves. Bad trade.
  Net combined: 21/28 → 15/28.
- **Where the budget hint went wrong**: maintainer-framing scenarios
  with multi-file refactors (the godot ones) genuinely need 15-25
  tool calls. The "20 is too many" guidance is calibrated for
  consumer-misuse / user-config scenarios (small fixes, 1-3 files)
  but applied uniformly to all bug classes. A maintainer scenario
  with 13 ground-truth files cannot be solved in 8 tool calls.
- **godot_4_2_world_environment** still fails (4 tool calls!) — the
  budget hint made it WORSE. The agent gave up before exploring.
  Confirms the forensic analysis: the agent has the wrong mental
  model entering the search; less time doesn't help, more time
  also doesn't (R12c/R13 had 50+ tool calls and same wrong answer).
  Reasoning-depth, not tool-count, is the bottleneck.
- **`godot_performance_on_android` flipped: solved both modes in
  R13 → solved code_only only in R14**. The with_bhdr version
  burned 41 tool calls and got file_level/judge=partial. R14's
  with_bhdr agent didn't get the budget benefit because it was
  the new Android-Java fix path (not browser-tier, so didn't get
  the GPA gate; budget hint didn't bite at 41).

## Open backlog

Carried forward to R15. Order by expected lift × inverse cost.

- **R15-P0 — refine budget hint by bug class.** The R14 hint applied
  uniformly to all bug classes was wrong. maintainer scenarios with
  13-file refactors need 15-25 tool calls; a "20 is too many" rule
  cuts them off. Move the hint to advisor + config_advice only (those
  are small fixes); remove from maintainer_framing. Expected lift:
  recovers most of R14's regression (+5/6 → +20-21/28 combined).
- **NEW: meta-hypothesis prompt** for "feature-toggle bugs"
  (forensic-godot_4_2_world_environment.md). When the user report
  identifies a specific feature toggle, look at the toggle's
  enabling path, not symptom-adjacent code. Untested; closes
  godot_4_2_world_environment if the hypothesis holds.
- **R12c-P3 — ensemble agents for borderline judges.** Could close
  godot_weird_shadow / godot_perf_on_android stochastic flips. ~$0.03/scenario.
- **NEW — review the cesium scenario.** It's failed in 4 consecutive
  rounds despite the agent producing a *correct* diagnosis (just for
  a different PR than mining picked). Either re-mine to capture
  PR #13098 as alternative, or accept agent's diagnosis as a
  separate verdict path.

## Side observations

- The browser-tier gate is the clearest win in three rounds: a
  single 50-line change closed a 3-solve gap. Worth pattern-matching
  similar "this whole class of scenarios doesn't benefit from this
  feature" gating decisions.
- Token-vs-solves tradeoff is now visible: R14 traded 6 solves for
  20% token reduction. That's a bad trade — but it shows the lever
  exists. A more carefully calibrated budget might recover both.
- Three consecutive rounds with zero system changes. The eval-
  pipeline-only lifts are running out. Next round needs to either:
  (a) ship a system change (e.g. WebGL backend), or
  (b) accept the current ceiling and shift focus to scenario quality
      (re-mining cesium-style multi-PR scenarios).
