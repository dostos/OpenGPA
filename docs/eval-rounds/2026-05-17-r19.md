## System-improvement audit (Musk 5-step)

**1. Requirements less dumb.**

- "Maintainer prompt always pushes 10-15 file investigation." Owner:
  me, R10 spec. Still load-bearing? **No** — R18 forensic showed it
  cost 4× tokens for +1 solved on a cohort where most fixes were
  single-file. Tiering by `len(fix.files)` makes the depth language
  match the actual canonical-fix size.
- "Mining's `bug_class` label is reliable enough to dispatch on."
  Owner: me, R10. Still load-bearing? **No, definitively** — R18
  showed 100% mis-classification on the mined cohort. R18-P2 deleted
  the dispatch; R19-P4 now actively *flags* the mis-labels in the
  verifier so future mining cycles can catch them.
- "`with_bhdr` mode is unreachable without a synthetic reproducer."
  Owner: me, R16 backlog. Still load-bearing? **Yes, but the
  inverse is provable now** — R19-P2 confirmed the *native pipeline*
  works end-to-end. The blocker is purely the cohort, not the
  product. Re-classified from "system bug" to "cohort-design gap"
  for R20.

**2. Deletions.**

- `expected_failure` block on world_environment (the flag was wrong)
- Redundant `bug_class == "framework-internal"` clause in
  `should_score` (replaced by `parsed_json` alone)
- "10-15 file" pitch as the *default* depth framing (replaced by
  tiered text that only fires on actual large fixes)

**3. Simplifications.**

- `_select_prompt_for_scenario` now passes `fix_files_count` to
  `render_prompt` — one dispatch axis (file count), reliably derived
  from `ScenarioMetadata.fix.files`.
- Mining-quality and bug_class-mismatch warnings share a single soft
  layer in `_check_mining_quality`; the verifier already had the
  warning-collection plumbing from R12c.

**4. Cycle-time improvements.**

- The native pipeline unblock means future "is the capture working?"
  smoke tests are now `bin/bhdr` + `LD_PRELOAD=libbhdr_gl.so ...`,
  no more hand-rebuilding under the old gpa names.

**5. Automation.**

- R19-P4 is automation: the soft-warning layer surfaces mining
  mis-classifications without manual review. 23 of 189 already
  flagged on the existing corpus.

## Open backlog (R20)

- **R20-P0: synthetic-reproducer slot in scenario.yaml.** The single
  biggest strategic gap remains: `with_bhdr` cannot run against
  mined scenarios. R18 round log proposed two paths; R19 confirms
  the native pipeline works on synthetic scenarios (E-series). The
  missing piece is: mining LLM produces a 200-line C reproducer
  alongside the fix metadata, the verifier ensures it compiles,
  the harness builds + runs it under the shim. Spec-worthy.
- **R20-P1: investigate cesium_camera_jumps mis-mining.** R19-P4
  now flags it. Multi-PR closures (the canonical anti-pattern) need
  a verifier rule: when a GitHub issue is closed by ≥2 PRs, prefer
  the most-files-changed PR or quarantine. Cheap to add.
- **R20-P2: dashboard build automation.** The dashboard rebuilds
  manually. Hook it into the eval CLI: after `bhdr.eval.cli run`
  saves results, auto-trigger `scripts/build-eval-dashboard.sh`.
  Cycle-time win.
- **R20-P3: vendor Plotly + marked.js.** Dashboard depends on
  CDN-hosted libraries. Vendor them under `dashboard/vendor/` so
  the dashboard works offline. R18 spec already mentioned this.
