# Round R19 (2026-05-17)

First post-rename round. The project is now **Beholder** (`bhdr`
namespace). Five changes shipped: P0 tiered-depth prompt, P1+P3
cleanups from the R18 backlog, P2 native-shim unblock, and P4
verifier mining-quality bar. The headline check is whether R19-P0
reverses the R17→R18 4× token-cost regression.

## Ran

- Cohort: same 13 scenarios as R17/R18 (web-map + godot)
- Modes: `code_only` (with_bhdr auto-skipped — current cohort is
  source-less so live capture has nothing to record against)
- Model: `claude-opus-4-7[1m]` via claude-cli
- Output: `/data3/bhdr-eval-results/2026-05-17-r19/code_only.json`
- Boundary: `round-r19-start` (commit `a6f38ce`)
- Variables vs R18:
  - **Eval pipeline (P0)**: maintainer prompt's depth language now
    tiers by `len(fix.files)`: 1-2 → "pointed edit", 3-9 → moderate,
    10+ → original deep framing.
  - **Eval pipeline (P1)**: `expected_failure` block removed from
    `godot_4_2_world_environment` — R18 proved it wasn't a stable
    failure.
  - **Eval pipeline (P3)**: redundant `bug_class == "framework-internal"`
    clause removed from `should_score`; missing JSON now uniformly
    falls through to prose/judge rather than auto-failing file_level.
  - **Eval pipeline (P4)**: verifier emits soft warning when
    `len(fix.files) >= 3` AND `bug_class != "framework-internal"`.
    23 of 189 scenarios flagged corpus-wide.
  - **System (P2)**: native shim + pybind module rebuilt under the
    `bhdr` namespace. Confirmed end-to-end: launcher creates
    socket+shm, shim connects, REST API serves draw-call data.
    Three rename-leftover bugs fixed (`bin/bhdr` exec'd `gpa.cli.main`,
    argparse prog="gpa", stale git worktrees blocking `bazel
    build //...`).

## Findings

- **R19-P0 (tiered prompt depth) is a regression.** The hope was
  "lower token cost on small-fix scenarios while preserving solves".
  The actual outcome: **−2 solved, ~0 token savings**. The 1-2 file
  "pointed edit" tier was too conservative — `vulkan_forward_severe`
  (R18's hero recovery) flipped from `file_level/high` to
  `no_signal/low` after being told "don't sprawl". The 10+ tier was
  too thin — `perf_on_android` slid from `judge` to `file_level/low`
  (failed), and `world_environment` slid `file_level/high` →
  `judge/medium` (qualified count dropped). R20 should revert P0
  unless an even smaller intervention can be isolated. The honest
  read: R18-P2's "blanket maintainer-prompt" was already close to
  optimal; cutting it doesn't help.
- **R19-P3 (drop redundant `bug_class` clause) is neutral.** Pre-R19
  the file scorer auto-fired on framework-internal scenarios without
  JSON, recording `solved=False, file_score=0.0`. Post-R19-P3 the
  same scenario now falls through to prose/judge. The R19 cohort
  produced one such transition (`vulkan_forward` ended at
  `no_signal` instead of file_level-failed). The verdict (failed)
  is the same; only the path differs. R19-P3 stays.
- **R19-P4 (bug_class mismatch warning) flagged 23 of 189 scenarios.**
  Quantifies R18's "mining mis-classifies 100% of framework bugs as
  consumer-misuse" forensic. Doesn't change eval signal directly —
  it's a verifier soft warning — but gives future mining cycles a
  catchable signal.
- **R19-P5 (sandbox) doesn't apply to this run.** The sandbox commit
  landed *after* the eval started; the running harness was the
  pre-sandbox path. R20 should re-run R18 cohort under the sandbox
  to detect if any R18/R19 solves depended on Ground Truth leakage.
- **R19-P2 (native pipeline unblocked) was the real win.** Three
  rename-leftover bugs fixed (`bin/bhdr` exec'd `gpa.cli.main`,
  argparse prog="gpa", stale worktrees blocking `bazel build`).
  End-to-end Beholder capture now works again: launcher creates
  socket+shm, shim connects, REST API serves draw-call data. Test
  scenarios still source-less so `with_bhdr` mode doesn't run in
  the eval, but the product side is back online.

## System-improvement audit (Musk 5-step)

**1. Requirements less dumb.**

- "Maintainer prompt always pushes 10-15 file investigation." Owner:
  me, R10 spec. Still load-bearing? **No** — R18 forensic showed it
  cost 4× tokens for +1 solved on a cohort where most fixes were
  single-file. Tiering by `len(fix.files)` makes the depth language
  match the actual canonical-fix size.
- "Mining's `bug_class` label is reliable enough to dispatch on."
  Owner: me, R10. Still load-bearing? **No, definitively** — R18
  showed 100% mis-classification on the mined cohort. R18-P2 deleted
  the dispatch; R19-P4 now actively *flags* the mis-labels in the
  verifier so future mining cycles can catch them.
- "`with_bhdr` mode is unreachable without a synthetic reproducer."
  Owner: me, R16 backlog. Still load-bearing? **Yes, but the
  inverse is provable now** — R19-P2 confirmed the *native pipeline*
  works end-to-end. The blocker is purely the cohort, not the
  product. Re-classified from "system bug" to "cohort-design gap"
  for R20.

**2. Deletions.**

- `expected_failure` block on world_environment (the flag was wrong)
- Redundant `bug_class == "framework-internal"` clause in
  `should_score` (replaced by `parsed_json` alone)
- "10-15 file" pitch as the *default* depth framing (replaced by
  tiered text that only fires on actual large fixes)

**3. Simplifications.**

- `_select_prompt_for_scenario` now passes `fix_files_count` to
  `render_prompt` — one dispatch axis (file count), reliably derived
  from `ScenarioMetadata.fix.files`.
- Mining-quality and bug_class-mismatch warnings share a single soft
  layer in `_check_mining_quality`; the verifier already had the
  warning-collection plumbing from R12c.

**4. Cycle-time improvements.**

- The native pipeline unblock means future "is the capture working?"
  smoke tests are now `bin/bhdr` + `LD_PRELOAD=libbhdr_gl.so ...`,
  no more hand-rebuilding under the old gpa names.

**5. Automation.**

- R19-P4 is automation: the soft-warning layer surfaces mining
  mis-classifications without manual review. 23 of 189 already
  flagged on the existing corpus.

## Open backlog (R20)

- **R20-P0 (NEW, top): revert R19-P0.** The tiered prompt traded 2
  solved scenarios for ~zero cost reduction. The R18 blanket
  maintainer prompt was closer to optimal. Restore the
  `## Task` body of `maintainer_framing.md` to the R18 version
  ("13-file refactor PRs are normal..."). Keep the rendering
  plumbing (`fix_files_count` param) — it doesn't cost anything if
  unused, and a future, smaller intervention might want it.
  Empirical lesson: cost-trimming requires harder evidence than
  "R17 was cheaper". The R17 cohort framing existed because R17's
  prompt LACKED depth pitch, not because it tiered it.
- **R20-P1: re-run R18 cohort under the R19-P5 sandbox.** The
  sandbox shipped after R19's eval started, so this round's
  numbers reflect the pre-sandbox (leaky) path. If sandbox-mode
  produces fewer solves than R18/R19, that's evidence prior
  rounds depended on Ground Truth leakage. Either result is
  informative: same numbers = leakage wasn't the binding factor;
  fewer numbers = need to harden the cohort.
- **R20-P2: synthetic-reproducer slot in scenario.yaml.** The
  biggest strategic gap remains: `with_bhdr` cannot run against
  mined scenarios. R19-P2 confirmed the native pipeline works on
  E-series (source-bearing) scenarios. Missing piece: mining LLM
  produces a minimal native reproducer alongside the fix metadata,
  verifier ensures it compiles, harness builds + runs under the
  shim. Spec-worthy.
- **R20-P3: investigate cesium_camera_jumps mis-mining.** R19-P4
  now flags multi-PR closures via the soft warning. Add a verifier
  rule: when a GitHub issue is closed by ≥2 PRs, prefer the
  most-files-changed PR or quarantine.
- **R20-P4: dashboard build automation.** Hook
  `scripts/build-eval-dashboard.sh` into the eval CLI's post-save
  step. Cycle-time win.
- **R20-P5: vendor Plotly + marked.js.** Dashboard depends on CDN-
  hosted libraries. Vendor under `dashboard/vendor/` for offline
  use. R18 spec already mentioned this.
