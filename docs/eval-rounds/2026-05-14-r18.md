## Findings

- **The bug_class prompt dispatch was theater.** Forensic on R17
  cohort: every mined scenario has `bug_class` of `consumer-misuse`
  or `user-config`, but `fix.files` contains 1-22 framework source
  files each. Mining was misclassifying 100% of scenarios. The
  agent already rebelled and emitted maintainer-style JSON anyway:
  5 of R17's 10 solves came through `file_level` scoring (which
  only parses `proposed_patches`), despite the advisor/config_advice
  prompts asking for `user_code_change` / `setting_change` schemas
  that nothing parses. The class-specific JSON tails had **zero
  downstream consumers** (grep confirms — only test files mention
  them). Audit step 2 deletion: collapse to one prompt.
- **All 3 stable-failure scenarios trace to the same root cause.**
  cesium_camera_jumps, world_environment, wrong_position_volumetric:
  all 3 are framework-internal bugs that mining classified as
  consumer-misuse or user-config. The mis-classification meant the
  prompt + scorer never aligned. Two of three (cesium 12 files,
  world_environment 13 files) had abundant fix.files signal — they
  *should* have scored via file_level. R18-P2's prompt change
  re-aligns them.
- **Token-spend pattern from R17 held.** Solved scenarios averaged
  ~5k output tokens; failed averaged ~9k. R17's pattern that
  "stuck-grind" scenarios cost ~2x is reproducible — the
  tool-call-budget signal in `Q4` from `docs/eval-next-steps.md`
  is real and worth shipping eventually.

scenario vs R17. The maintainer prompt's "13-file refactor PRs are
normal" framing pushes deeper investigation. Net cost/solve went
from 6,378 to 22,688 output tokens. The R17 outline-recipe deletion
got cheaper investigation; R18-P2 trades that off for higher signal
quality (file_level vs judge/prose) and +1 solve. **R19 must
address this** — either cap investigation depth or scope the
maintainer prompt's depth-prescription to actual fix.files cardinality
via the existing scope_hint (R12c P1 lever).

**The expected_failure call for world_environment was wrong.** R17
analysis attributed it to "reasoning-shallow at this model tier (5
rounds, same failure mode)." Wrong — it was prompt-mis-routing.

the agent solved it with high-confidence file_level scoring on its
first try. Lesson: stable failures should be classified only AFTER
ruling out pipeline misconfiguration. R18-P2's deletion did exactly
that ruling-out. Updating its expected_failure block in R19.

## Open backlog (R19)

- **R19-P0 (NEW): rein in maintainer-prompt investigation depth.**
  R18 trade +1 solved for 4x token cost. The "10-15 files / 13-file
  refactor" framing in `maintainer_framing.md` is the culprit. Two
  options: (a) tier the prompt by `len(fix.files)` (1-file fix gets
  "investigate the one file thoroughly", 10+-file fix gets the
  current depth pitch); (b) bind the depth language to the
  scope_hint string (already computed). Pick (a) — simpler, doesn't
  require prompt-time computation, doesn't risk leaking fix-file
  count to the agent as a soft hint.
- **R19-P1: remove `world_environment` from expected_failure list.**
  R18 proved it's not a stable failure — it was prompt-mis-routing.
  Keep the slot in scenario.yaml for the future; just drop the
  flag.
- **R19-P2: re-enable with_gla mode** — strategic-direction Q1.
  Current cohort is 100% source-less → with_gla auto-skips → the
  product's whole reason for existing (frame capture) hasn't been
  measured in 5+ rounds. Two paths:
  (a) **Synthetic-reproducer slot in scenario.yaml**: mining
      generates a minimal native repro alongside the mined fix.
      Larger investment, real measurement signal.
  (b) **Upstream-demo capture**: for frameworks that ship runnable
      demos (godot, three.js examples), run a demo under the shim
      and use that as the captured frame. Medium investment;
      doesn't directly anchor to the mined bug but at least
      exercises the capture pipeline.
  Pick one in R19. Without this, the rule "every round ships a
  system change" can't be satisfied honestly.
- **R19-P3: drop the `bug_class == "framework-internal"` clause
  from `should_score`** in harness.py. With R18-P2 the agent always
  gets the maintainer prompt → always asked for JSON → `parsed_json`
  alone is sufficient. The class check is now redundant. Cheap
  cleanup, ~3 lines.
- **R19-P4: mining-quality bar for bug_class** — deferred from
  R18-P1 backlog. Add a verifier check: if `len(fix.files) >= 3`
  AND `bug_class != "framework-internal"`, warn (mining likely
  mis-classified). Soft signal; doesn't block. Lower priority now
  that the prompt-dispatch deletion removed the consequence of
  mis-classification.
- **R19-P5: multi-PR closure detection** — see cesium_camera_jumps
  expected_failure note. When a GitHub issue is closed by ≥2 PRs,
  mining should warn or pick the most-files-changed PR.

  a similar pattern at a structural level: the per-class prompt
